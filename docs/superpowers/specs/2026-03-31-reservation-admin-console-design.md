# 预约管理端控制台设计

## 文档说明

- 文档日期：2026-03-31
- change 名称：`reservation-admin-console`
- 基线原则：以当前代码现状为准，不以旧文档为准
- 文档目标：补齐预约模块的管理端独立控制台，建立管理端查询、查看和取消预约的最小闭环

## 背景

当前预约模块的后端能力和用户端主流程已经存在：

- 后端已有用户创建预约、取消预约、查询我的预约、查看预约详情、检查时间冲突等接口
- 用户端已有预约列表、创建页、详情页等基础页面
- 管理端目前没有独立的预约控制台

当前管理端最接近预约能力的地方，是用户详情抽屉中的预约子表。但这只是“按用户附带查看”，不能满足管理员按全局维度查询、筛选、查看和处理预约的需求。

因此这轮不做预约规则重构，而是补齐管理端控制台。

## 目标

本轮要实现以下目标：

1. 后端提供管理员预约列表接口
2. 后端提供管理员预约详情接口
3. 后端提供管理员取消预约接口
4. 管理端提供独立的预约列表页和详情页
5. 管理员可以筛选、查看并取消 `PENDING` 状态预约
6. 文档与 OpenSpec 同步更新

## 范围

### 本轮包含

- 后端管理员预约列表接口
- 后端管理员预约详情接口
- 后端管理员取消预约接口
- 管理端预约列表页
- 管理端预约详情页
- 列表筛选、分页、查看详情、取消预约
- 对应 API、类型、store、router、导航入口
- 基线文档、交接文档、OpenSpec 文档同步

### 本轮不包含

- 用户端预约流程重做
- 预约创建规则改造
- 预约过期任务重写
- Redisson 并发锁方案重构
- 预约转充电联动规则调整
- 预约消息提醒或实时推送

## 方案对比

### 方案一：独立管理控制台

新增管理员预约接口，并在 `frontend-admin` 中新增独立预约控制台。

优点：

- 边界清晰
- 与充电记录、故障报修的管理端治理方式一致
- 便于按全局维度检索和排查预约

缺点：

- 需要补一组完整的 admin API、类型和页面

### 方案二：继续扩展用户详情抽屉

在用户详情抽屉的预约子表基础上继续扩展详情和取消能力。

优点：

- 改动较小

缺点：

- 不能形成独立控制台
- 无法满足全局查询和筛选
- 会继续把预约管理耦合在用户管理模块中

### 方案三：只补后端接口

先做管理员预约列表、详情、取消接口，前端后续再补页面。

优点：

- 单次改动更小

缺点：

- 当前交付不完整
- 不能解决管理端缺控制台的问题

### 推荐方案

采用方案一：独立管理控制台。

原因：

- 当前缺口本质上是管理端独立入口缺失，而不是某个单点接口缺失
- 该方案最符合当前项目的模块治理策略
- 可在不重构用户端流程的前提下，补齐管理端闭环

## 后端设计

### 接口设计

新增管理员预约接口：

- `GET /admin/reservations`
- `GET /admin/reservations/{id}`
- `PUT /admin/reservations/{id}/cancel`

### 查询能力

管理员列表支持以下筛选条件：

- 预约状态
- 用户 ID 或用户名关键字
- 充电桩 ID 或充电桩编号关键字
- 开始时间范围
- 结束时间范围
- 分页参数

如果现有实体字段和仓库查询方式不足以支持全部筛选，可以先落成“状态 + 用户 + 充电桩 + 时间范围”的最小组合，不在这一轮扩展复杂搜索 DSL。

### 管理员取消规则

管理员只能取消 `PENDING` 状态预约。

取消逻辑要求：

1. 读取预约记录
2. 校验预约状态必须为 `PENDING`
3. 将预约状态改为 `CANCELLED`
4. 如果对应充电桩当前状态仍为 `RESERVED`，则恢复为 `IDLE`
5. 返回统一业务结果

这里要避免粗暴恢复充电桩状态。实现上应至少基于当前预约记录和当前充电桩状态做安全判断，避免误改其他占用场景。

### 服务层设计

`ReservationService` 增加管理端能力：

- `getAdminReservations(...)`
- `getAdminReservationById(...)`
- `cancelReservationByAdmin(...)`

实现优先复用现有 `ReservationResponse`，避免额外制造第二套预约响应模型。如果管理端确实需要更多展示字段，再只补最小增量字段。

### 数据访问

`ReservationRepository` 可以补充以下方向的能力：

- 管理端分页查询
- 按条件组合筛选
- 必要时配合 `Specification` 或 JPQL 完成时间范围和状态过滤

这一轮的目标是完成管理端控制台，不要求顺手做仓储层通用化重构。

## 前端管理端设计

### 新增文件

- `frontend-admin/src/api/reservation.ts`
- `frontend-admin/src/types/reservation.ts`
- `frontend-admin/src/stores/reservation.ts`
- `frontend-admin/src/views/admin/ReservationList.vue`
- `frontend-admin/src/views/admin/ReservationDetail.vue`

### 需要更新的文件

- `frontend-admin/src/router/index.ts`
- `frontend-admin/src/App.vue`

### 页面设计

#### 预约列表页

列表页负责：

- 加载预约分页数据
- 展示状态、用户、充电桩、预约时间等信息
- 提供筛选项
- 支持跳转详情
- 对 `PENDING` 预约提供取消操作

筛选区建议包含：

- 状态下拉
- 用户关键字
- 充电桩关键字
- 时间范围

#### 预约详情页

详情页负责展示：

- 预约编号
- 用户信息
- 充电桩信息
- 开始时间
- 结束时间
- 状态
- 创建时间
- 更新时间

如果状态为 `PENDING`，详情页也允许执行取消操作。

### 类型设计

优先从现有用户管理抽屉的预约字段中提取共用结构，避免让预约类型继续挂在用户模块下面。

推荐建立：

- `ReservationStatus`
- `ReservationItem`
- `ReservationPageParams`
- `AdminReservationQuery`

目标是让预约控制台拥有独立类型边界，而不是继续依赖 `user.ts` 中的子表类型。

### Store 设计

管理端预约 store 负责：

- 保存预约列表
- 保存分页信息
- 保存当前预约详情
- 执行列表查询
- 执行详情查询
- 执行管理员取消

它不负责用户端预约逻辑，也不和 `frontend-client` 共享行为状态。

## 兼容与回归策略

本轮不移除用户详情抽屉中的预约子表。

兼容策略如下：

- 用户详情抽屉预约子表继续保留
- 新增独立预约控制台作为正式管理入口
- 如果可以复用公共类型则复用，不强行复用 UI 逻辑

这样可以确保：

- 用户管理模块已有查看能力不被破坏
- 管理员获得更适合全局检索的入口

## 风险

### 充电桩状态联动风险

管理员取消预约会同步修改充电桩状态，如果判断过粗，可能把不应恢复的状态误置为 `IDLE`。

### 类型漂移风险

用户详情抽屉已经存在预约子表，若这轮直接复制一套结构，后续容易再次产生类型漂移。

### 范围膨胀风险

预约模块还涉及过期任务、时间冲突、转充电联动等复杂逻辑。这些都不应混入本轮，否则 change 会失控。

## 验证方案

### 后端验证

- 管理端预约列表在常见筛选下返回正确
- 管理端详情接口返回正确
- 管理员取消 `PENDING` 预约成功
- 管理员取消非 `PENDING` 预约失败
- 取消后充电桩状态正确恢复

### 前端验证

- 管理端列表页可正常加载
- 状态、用户、充电桩、时间范围筛选正常
- 分页正常
- 详情页展示正常
- 列表页和详情页中的取消操作正常

### 回归验证

- 用户端我的预约、预约详情、用户取消不受影响
- 用户管理抽屉的预约子表仍然正常

## 实施顺序

1. 后端补管理员预约列表、详情、取消接口
2. 管理端补预约 API、类型、store
3. 管理端补列表页、详情页、导航入口
4. 更新 OpenSpec、基线文档、总盘点文档和交接文档
5. 执行编译与类型检查验证

## 最终结论

`reservation-admin-console` 的定位，是在不重构用户端预约流程的前提下，补齐预约模块的管理端独立控制台。

这一轮只做：

- 管理端列表
- 管理端详情
- 管理端取消 `PENDING`

不做预约业务规则重构。这样可以在颗粒度可控的前提下，把预约模块从“用户端已实现、管理端明显缺口”推进到更完整的后台闭环。
