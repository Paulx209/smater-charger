# 预警通知管理端对齐设计

## 文档说明

- 文档日期：2026-03-31
- change 名称：`warning-notice-admin-alignment`
- 基线原则：以当前代码现状为准，不以旧文档为准
- 文档目标：补齐预警通知模块的管理端独立控制台，建立列表、详情、批量已读、删除和全局阈值设置的最小闭环

## 背景

当前预警通知模块已经具备用户侧主流程：

- 后端已有用户查询预警通知列表、未读数、标记已读、删除和阈值配置接口
- 用户端已有通知列表、设置页和未读数展示
- 管理端目前没有独立的预警通知控制台

因此当前缺口不在于“预警通知不存在”，而在于“管理端缺少独立入口与全局视角”。

## 目标

本轮要实现以下目标：

1. 后端提供管理员预警通知列表接口
2. 后端提供管理员预警通知详情接口
3. 后端提供管理员批量标记已读接口
4. 后端提供管理员删除通知接口
5. 后端提供全局预警阈值读取与更新接口
6. 管理端提供独立预警通知列表、详情与设置页
7. 文档与 OpenSpec 同步更新

## 范围

### 本轮包含

- 后端管理员预警通知列表接口
- 后端管理员预警通知详情接口
- 后端管理员批量标记已读接口
- 后端管理员删除接口
- 后端全局预警阈值读取与更新接口
- 管理端预警通知列表页
- 管理端预警通知详情页
- 管理端全局阈值设置页
- 对应 API、类型、store、router、导航入口
- 基线文档、交接文档、OpenSpec 文档同步

### 本轮不包含

- 用户端通知中心重做
- 通知生成规则重构
- 短信或 WebSocket 推送改造
- 用户级阈值策略重构
- 充电超时检测任务重写

## 方案对比

### 方案一：独立管理控制台

新增管理员预警通知接口，并在 `frontend-admin` 中新增独立控制台。

优点：

- 边界清晰
- 与预约、充电记录、故障报修的管理端治理方式一致
- 能形成全局视角的通知管理入口

缺点：

- 需要补一组完整的 admin API 和页面

### 方案二：继续依附现有用户或统计页面

在已有页面中零散增加预警通知能力。

优点：

- 改动较小

缺点：

- 缺少独立入口
- 预警通知管理会继续分散
- 不利于后续维护和排查

### 方案三：只补后端接口

这轮只做管理员接口，页面后续再补。

优点：

- 单次改动更小

缺点：

- 交付不完整
- 不能解决管理端缺入口的问题

### 推荐方案

采用方案一：独立管理控制台。

原因：

- 当前缺口本质上是管理端独立入口缺失，而不是单个接口缺失
- 该方案最符合当前项目的模块治理策略
- 可以在不干扰用户端的前提下，补齐管理端闭环

## 后端设计

### 接口设计

新增管理员预警通知接口：

- `GET /admin/warning-notices`
- `GET /admin/warning-notices/{id}`
- `PUT /admin/warning-notices/read`
- `DELETE /admin/warning-notices/{id}`
- `GET /admin/warning-notices/config/threshold`
- `PUT /admin/warning-notices/config/threshold`

### 查询能力

管理员列表支持以下筛选条件：

- 通知类型
- 已读状态
- 用户 ID
- 时间范围
- 分页参数

这一轮优先满足“类型 + 已读状态 + 用户 + 时间范围”的最小组合，不扩展复杂检索语法。

### 全局阈值设计

本轮新增的是“全局预警阈值”，与用户个人阈值分离。

要求：

- 读取接口返回当前全局阈值
- 更新接口只修改系统级配置
- 不覆盖用户个人阈值

### 服务层设计

`WarningNoticeService` 增加管理端能力：

- `getAdminWarningNoticeList(...)`
- `getAdminWarningNoticeById(...)`
- `markAdminWarningsAsRead(...)`
- `deleteAdminWarningNotice(...)`
- `getAdminThresholdConfig()`
- `setAdminThresholdConfig(...)`

优先复用现有 `WarningNoticeResponse`，避免制造第二套通知响应模型。

### 数据访问

`WarningNoticeRepository` 视需要补充：

- 管理端分页查询
- 已读状态与类型筛选
- 时间范围筛选

如果系统配置仓库已有承载阈值的模式，则应直接复用，不单独造新的存储表。

## 前端管理端设计

### 新增文件

- `frontend-admin/src/api/warningNotice.ts`
- `frontend-admin/src/types/warningNotice.ts`
- `frontend-admin/src/stores/warningNotice.ts`
- `frontend-admin/src/views/admin/WarningNoticeList.vue`
- `frontend-admin/src/views/admin/WarningNoticeDetail.vue`
- `frontend-admin/src/views/admin/WarningNoticeSettings.vue`

### 需要更新的文件

- `frontend-admin/src/router/index.ts`
- `frontend-admin/src/App.vue`

### 页面设计

#### 预警通知列表页

列表页负责：

- 加载通知分页数据
- 展示通知类型、内容摘要、用户、时间、已读状态
- 提供筛选项
- 提供批量已读操作
- 提供删除操作
- 支持跳转详情

筛选区建议包含：

- 通知类型
- 已读状态
- 用户 ID
- 时间范围

#### 预警通知详情页

详情页负责展示：

- 通知编号
- 通知类型
- 通知内容
- 所属用户
- 已读状态
- 创建时间
- 关联记录信息（如果存在）

详情页不必承担复杂编辑行为，重点是查看和删除。

#### 全局阈值设置页

设置页负责：

- 读取全局预警阈值
- 更新全局预警阈值
- 清晰区分“系统级阈值”和“用户个人阈值”

## 类型设计

管理端应建立独立的通知类型边界，不直接复用 `frontend-client` store。

推荐建立：

- `WarningNoticeType`
- `WarningNoticeInfo`
- `WarningNoticeQueryParams`
- `WarningNoticePage`
- `AdminThresholdConfig`

用户端已有通知类型映射可以参考，但不直接共享管理端状态实现。

## Store 设计

管理端预警通知 store 负责：

- 保存通知列表
- 保存分页信息
- 保存当前通知详情
- 执行列表查询
- 执行批量已读
- 执行删除
- 读取和更新全局阈值

它不负责用户端未读数展示，也不与 `frontend-client` 共享行为状态。

## 兼容与回归策略

本轮不移除用户端通知列表和设置页。

兼容策略如下：

- 用户端通知能力继续保留
- 新增独立管理端控制台作为正式后台入口
- 全局阈值与用户阈值分开处理

这样可以确保：

- 用户端主流程不受破坏
- 管理员获得全局通知管理能力
- 阈值配置职责不再混乱

## 风险

### 阈值概念混淆风险

系统级阈值和用户级阈值都存在，如果接口边界不清晰，容易互相覆盖。

### 通知结构差异风险

不同类型的通知可能拥有不同的关联字段，详情页不能假设所有通知结构一致。

### 权限边界风险

如果直接复用用户侧接口语义，可能把“用户自己的通知操作”和“管理员的全局管理”混在一起。

## 验证方案

### 后端验证

- 管理端通知列表在常见筛选下返回正确
- 管理端详情接口返回正确
- 管理员批量已读成功
- 管理员删除通知成功
- 全局阈值读取与更新成功

### 前端验证

- 管理端列表页可正常加载
- 类型、已读状态、用户、时间范围筛选正常
- 批量已读正常
- 删除正常
- 详情页展示正常
- 阈值设置页可正常读取与更新

### 回归验证

- 用户端通知列表、未读数、标记已读、删除能力不受影响
- 用户端阈值设置不被误改为系统级阈值逻辑

## 实施顺序

1. 后端补管理员预警通知列表、详情、批量已读、删除、阈值接口
2. 管理端补预警通知 API、类型、store
3. 管理端补列表页、详情页、设置页、导航入口
4. 更新 OpenSpec、基线文档、总盘点文档和交接文档
5. 执行编译与类型检查验证

## 最终结论

`warning-notice-admin-alignment` 的定位，是在不重构用户端通知流程的前提下，补齐预警通知模块的管理端独立控制台。

这一轮只做：

- 管理端列表
- 管理端详情
- 管理端批量已读
- 管理端删除
- 全局阈值设置

不做通知生成和推送机制重构。这样可以在颗粒度可控的前提下，把预警通知模块从“用户端已实现、管理端明显缺口”推进到更完整的后台闭环。
