# 项目模块基线识别 Spec

## 目的

这份 spec 用来固定当前项目的代码基线，解决“代码、旧文档、OpenSpec 漂移”的问题。

它不描述一个未来功能，而是描述当前仓库在 `2026-03-30` 的真实状态，用于后续所有 OpenSpec change 的起点。

## 范围

本次识别覆盖以下一级业务模块：

- 用户与认证
- 车辆管理
- 预约管理
- 充电桩管理
- 充电记录
- 系统公告
- 费用配置
- 故障报修
- 预警通知
- 统计与看板

对照范围：

- `backend`
- `frontend-admin`
- `frontend-client`
- `接口文档设计/`
- `openspec/`

## 判定原则

1. 代码优先，文档次之
2. 有文件不等于已闭环，必须看路由、接口、store、页面或控制器是否形成链路
3. 前后端存在明显契约错位时，判定为 `部分实现`，而不是 `已实现`
4. OpenSpec 只用于后续变更规划，不再被当作“当前系统全貌”

## 关键发现

### 1. 旧文档大面积过时

旧的 `接口文档设计/待实现功能清单.md` 与 `接口文档设计/开发路线规划.md` 仍把以下模块视为“待开发”：

- 费用配置
- 充电记录
- 预警通知
- 故障报修
- 公告
- 用户管理
- 数据统计

但这些模块在代码中已经全部出现了后端实现，且其中一部分已经具备完整前端能力。

### 2. 当前真正的风险不是“没写代码”，而是“闭环不真实”

目前更严重的问题不是空白模块，而是以下三类偏差：

- 旧文档仍按“未实现”描述已经存在的模块
- 部分前端接口契约与后端真实路径不一致
- 管理端能力有一部分被混放在车主端工程中

### 3. 系统公告模块的契约问题已完成校正

`announcement-contract-alignment` 已完成以下收敛：

- `frontend-admin` 与 `frontend-client` 的公告 API 路径已改为资源路径，不再重复拼接 `/api`
- `frontend-client` 中混入的 `/admin/announcement` 路由与页面已移除
- 管理公告能力重新收敛到 `frontend-admin`

当前剩余工作不在公告契约本身，而在项目范围内已有的存量类型错误和后续运行时联调。

### 4. 故障报修模块存在明确契约错位

车主端故障报修代码使用的是：

- `/api/fault-reports`
- `/api/fault-reports/my`
- `/api/fault-reports/upload`
- `/api/fault-reports/stats/pile/{id}`

而当前后端控制器实际暴露的是：

- `/fault-report`
- `/fault-report/admin/all`
- `/fault-report/admin/{id}`
- `/fault-report/admin/{id}/handle`
- `/fault-report/admin/statistics`

这说明故障报修模块目前不是“功能缺一点”，而是“契约还没有统一”。

### 5. 车主端混入了管理能力

`frontend-client` 当前包含：

- `/admin/price-config`
- `/admin/announcement`

这会造成以下问题：

- 管理能力入口分散
- 模块边界不清晰
- 后续 OpenSpec 不容易按端拆分

## 对 OpenSpec 的判断

当前仓库中最完整的 change 是：

- `openspec/changes/charging-record-frontend`

判定：

- 该 change 与代码现实基本一致，说明单模块 OpenSpec 工作流是可用的
- 但它不能覆盖整个项目现状
- 项目目前缺少“代码基线层”，所以才导致后续文档继续失真

结论：

- OpenSpec 继续保留
- 但从现在开始，任何新 change 都必须建立在这份基线识别结果之上

## 推荐的下一步 change

### 推荐 1：`fault-report-contract-alignment`

范围：

- 统一故障报修前后端路径、分页和详情契约
- 删除或补齐当前前端假设但后端不存在的接口

原因：

- 当前不是纯 UI 缺口，而是链路实际不通

### 推荐 2：`admin-charging-record-console`

范围：

- 基于已存在的后端 `/charging-record/admin/all`
- 建立管理端独立充电记录页

原因：

- 属于典型“后端现成、管理端缺口明确”的模块

## 输出结果

本次识别完成后，项目应形成三层文档结构：

1. 代码事实层
   - 本 spec 和模块盘点文档承担这层职责
2. 项目总览层
   - `接口文档设计/待实现功能清单.md`
3. 变更规划层
   - `openspec/changes/...`

后续协作规则：

- 先确认模块基线
- 再选一个模块
- 再建立一个窄范围 change
- 最后按 tasks 实现
